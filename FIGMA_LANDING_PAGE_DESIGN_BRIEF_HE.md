# הנחיות מקצועיות לסוכן FIGMA - דף נחיתה RSVP

## מטרה
לעצב דף נחיתה RSVP לאירוע כחוויה רציפה, אלגנטית וחמה, בשפה ויזואלית נקייה ומדויקת שממירה לפעולת אישור הגעה.

הדף מיועד לאורח קצה שמגיע מקישור אישי לאירוע, ממלא פרטים, בוחר סטטוס הגעה ושולח אישור.

---

## חוקי עיצוב קשיחים (חובה)
יש להשתמש ב-3 צבעים בלבד בכל המסך:

- רקע ראשי: `#FFFFFF` (לבן)
- צבע דומיננטי לדגשים/כותרות/מצבי בחירה/כפתור ראשי: `#FF1A5F` (פוקסיה)
- צבע ניטרלי לגבולות/קווי הפרדה/מצב רגיל של שדות: `#D9D9D9` (אפור בהיר)

אין להוסיף צבע רביעי (כולל שחור/כהה). טקסטים רגילים יוצגו בפוקסיה או באפור בהתאם להיררכיה.

---

## שפה עיצובית וחוויית משתמש
- חוויית מסך רציפה (Flow), לא "בלוקים טכניים" מופרדים מדי.
- מבנה אנכי נקי עם ריווח נדיב, תחושת אירוע חגיגית ולא טופס בירוקרטי.
- RTL מלא (עברית), יישור לימין לטקסטים, עם היררכיה טיפוגרפית ברורה.
- רוחב תוכן מומלץ בדסקטופ: 680-760px ממורכז; במובייל: 100% עם padding צדדי 20-24px.
- קצב ריווח מומלץ: 8/12/16/24/32 לפי היררכיה.

## תוכן וקומפוזיציה (Top to Bottom, בצורה זורמת)

### פתיח אירוע
כותרת ראשית גדולה בפוקסיה:

`הנכם מוזמנים לאירוע {{סוג אירוע}} של {{שמות}}`

מיד מתחת לכותרת יש להציג את פרטי האירוע בצורה נגישה וברורה עם אייקונים/בולטים בפוקסיה:
- מתחם
- עיר
- רחוב ומספר
- תאריך
- שעה

בסיום אזור הפתיח: קו הפרדה דק בגוון `#D9D9D9`.

### אזור מילוי פרטי אורח
שדות הקלט:
- שם מלא
- טלפון
- כמות מגיעים

חוקי UI לשדות:
- רקע שדה לבן (`#FFFFFF`)
- מסגרת ברירת מחדל אפורה (`#D9D9D9`)
- ב-focus המסגרת הופכת לפוקסיה (`#FF1A5F`)
- פינות מעוגלות עדינות (8-12px) וגובה שדה נוח למגע (מינימום 44px)

### בחירת סטטוס הגעה
3 כפתורי סטטוס גדולים וברורים (אנכי במובייל, אופציונלית אופקי בדסקטופ):
- מגיע
- לא מגיע
- אולי יגיע (טקסט תצוגה)

חוקי מצב:
- מצב רגיל: רקע לבן, מסגרת אפורה
- מצב נבחר: הדגשת פוקסיה ברורה (רקע מלא פוקסיה או טקסט+מסגרת פוקסיה)
- חייב להיות ברור במבט ראשון איזה סטטוס נבחר

### סיום והנעה לפעולה
- כפתור ראשי גדול: `אישור הגעה`
- צבע רקע הכפתור: פוקסיה (`#FF1A5F`)
- טקסט הכפתור: לבן (`#FFFFFF`)
- מתחת לכפתור: טקסט סיום חם בפוקסיה:
  `נשמח לראותכם בין אורחינו...`

## מיפוי מדויק למערכת (Data Contract) - חובה להתאים לשמות בפועל

### נתוני אירוע שמגיעים מהשרת (Read)
Endpoint:
- `GET /api/public/event/:eventId`

מבנה התשובה:
- `eventId`
- `event` אובייקט עם השדות:
  - `eventType` -> סוג אירוע (למשל: חתונה/ברית/אחר)
  - `eventNames` -> שמות המארחים/הזוג
  - `venueName` -> שם המתחם
  - `city` -> עיר
  - `streetAndNumber` -> רחוב ומספר
  - `eventDate` -> תאריך אירוע (string)
  - `eventTime` -> שעת אירוע (string)

התאמה לתוכן בדף:
- `{{סוג אירוע}}` = `eventType`
- `{{שמות}}` = `eventNames`
- מתחם = `venueName`
- עיר = `city`
- רחוב = `streetAndNumber`
- תאריך = `eventDate`
- שעה = `eventTime`

### נתוני RSVP שנשלחים מהדף (Write)
Endpoint:
- `POST /api/public/event/:eventId/rsvp`

Body חובה/רלוונטי:
- `fullName` (חובה)
- `phone` (חובה)
- `attendeesCount` (מספר; ברירת מחדל 1)
- `status` (חובה)

ערכי `status` המותרים בפועל בשרת:
- `מגיע`
- `לא מגיע`
- `אולי`

הערה חשובה ל-FIGMA:
- בטקסט המוצג למשתמש אפשר לכתוב "אולי יגיע", אבל הערך הלוגי שנשלח חייב להיות `אולי`.

- Input: default / focus / error
- Status button: default / selected / pressed
- Submit button: default / loading / success
- לאחר שליחה מוצלחת: מסר אישור חיובי ברור באותה שפה עיצובית

## מסכי חובה למסירה מהמעצב
- מסך מובייל מלא (Primary)
- מסך דסקטופ מלא
- סט קומפוננטים עם וריאנטים (שדות, כפתורי סטטוס, כפתור שליחה)
- Spec מרווחים/גדלים/טקסטים ל-handoff

## קריטריוני קבלה
- שימוש ב-3 צבעים בלבד לפי ההגדרות
- חוויית Flow רציפה וחגיגית (לא חלוקה טכנית יבשה)
- התאמה מלאה לשמות השדות של המערכת ללא שינויי naming
- מצב בחירה ברור מאוד לכפתורי סטטוס
- CTA מרכזי ובולט להשלמת אישור הגעה
